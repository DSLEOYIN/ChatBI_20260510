# ChatBI Workspace 2.0 产品需求文档（PRD 草案）

> 状态：头脑风暴草案  
> 日期：2026-05-19  
> 来源：基于现有需求调研、原型验收、动态卡片讨论、LangGraph 开发规格与 Dify/API 集成文档整理  

## 1. 产品定位

ChatBI Workspace 2.0 是面向广汽国际经营层、业务团队与数据技术团队的企业级 AI 数据智能平台。用户通过自然语言提出经营问题，系统由 ReAct Agent 自主规划任务、选择 Skill、调用 MCP 工具，并最终把用户可见的回答、推理执行过程、图表、表格和结论全部呈现在业务画布上。

产品核心定位为：AI 智能出数 + 动态 HTML 业务画布 + 深度经营分析报告。

第一阶段必须以现有前端原型 `前端原型/index_v3_dynamic.html` 为体验基准，所有后端接口先做 mock，确保前端交互、Agent 思维链/执行链流式展示、画布渲染、下载等流程全部跑通，再逐步替换为真实 MySQL、Dify、MCP 和大模型接口。

## 2. 核心问题

1. 写经营汇报和 PPT 时，业务人员大量时间消耗在找数、取数、整理图表上。
2. 传统 BI 门槛高，非技术用户需要理解维度、指标、表结构和拖拽建模。
3. ETL 与数据团队被大量临时取数需求占用，业务响应慢。
4. 用户难以信任 AI 查数过程，需要在画布上看到 Agent 思维链/执行链、工具调用、SQL、口径解释和数据来源。
5. 固定看板无法适应闲聊、简单查询、复杂分析、联网搜索、指标解释等不同意图，右侧需要由最终回答结果驱动生成不同 HTML 内容。

## 3. 目标用户

| 用户类型 | 典型诉求 | 使用场景 |
| --- | --- | --- |
| 经营层 | 快速看大盘、风险预警、拿到决策建议 | 月度经营会、区域复盘、库存专题 |
| 业务负责人 | 查询销量、库存、订单、排产、目标达成 | 日常经营分析、临时汇报材料 |
| 数据/IT 团队 | 降低取数工单、管理口径和权限 | 数据服务治理、知识库维护 |
| 产品/运营人员 | 用自然语言生成图表和结论 | 原型演示、业务培训 |

## 4. 产品目标

### 4.1 MVP 目标

1. 以后端 mock 接口跑通 `index_v3_dynamic.html` 已设计的完整快速问答流程。
2. 支持用户通过悬浮 AI 助手提出问题，并在画布上流式展示大模型回答与 Agent 思维链/执行链。
3. 支持快速问答模式：生成 mock SQL、mock 查询结果、简短分析、HTML 画布内容。
4. 支持会话历史、新对话、历史删除和历史恢复。
5. 支持动态画布，根据 Agent 最终产物展示不同 HTML 内容。
6. 支持 Agent 基于问题自主选择 Skill，再由 Skill 调用对应 MCP 工具。

### 4.2 非目标 / 本期不做

1. 不实现“导数助手”Tab。
2. 不做完整自助 BI 拖拽建模。
3. 不开放用户任意写 SQL 执行能力。
4. 不把 Dify 作为主工作流编排引擎，Dify 仅作为知识库检索来源。
5. 不支持用户编辑或删除右侧画布组件。
6. 第一阶段不直连真实 MySQL，先用 mock 数据跑通全链路。
7. MVP 不实现深度思考长报告，只实现快速问答。
8. MVP 暂不考虑数据权限问题。

## 5. 核心用户旅程

### 5.1 大盘概览

用户输入：“帮我看一下本月中东公司销量和目标达成情况。”

系统应返回：
1. Agent 自主判断需要完成的任务。
2. Agent 选择合适 Skill，例如销售分析 Skill。
3. Skill 内部声明可用流程和 MCP 工具，例如 Dify 知识库 MCP、MySQL 查询 MCP、Tavily 搜索 MCP。
4. Agent 按实时推理结果调用一个或多个工具，不要求每次链路完全一致。
5. 画布展示用户可见回答、执行链、SQL/工具结果摘要、KPI、图表、明细表和结论建议。

### 5.2 库存预警

用户输入：“诊断中东 GS8 库存是否有爆库风险。”

系统应返回：
1. 展示库存天数、在店库存、在途库存、终端销售趋势。
2. 标记风险等级。
3. 分析库存异常可能原因。
4. 给出去库存建议。

### 5.3 经营分析长报告（MVP 后）

用户切换“深度思考”后输入：“生成一份中东公司 GS8 库存经营分析报告。”

系统应返回：
1. 结构化报告，包括现状、原因拆解、同比/环比、年度达成、策略建议。
2. 支持引用查询 SQL、指标口径、数据来源。
3. 右侧 HTML 画布展示报告重点、图表和明细数据。
4. 明细数据可供用户下载。

## 6. 功能需求

### 6.1 对话入口

1. 提供右下角悬浮入口。
2. 支持精简悬浮窗和全屏工作区两种状态。
3. 精简状态下提问不应自动全屏。
4. 支持新对话、关闭、展开、历史入口。
5. 支持输入框自适应高度。

### 6.2 ReAct Agent 自主规划

系统不应把后端链路写死成固定节点流程，而应由 ReAct Agent 根据用户问题动态规划：

1. 相同问题在不同上下文下，可能产生不同思考路径和工具调用顺序。
2. Agent 先理解问题，再自主选择合适 Skill。
3. Skill 负责详细描述适用场景、执行流程、可调用 MCP 工具、输入输出约束。
4. Agent 根据 Skill 描述决定是否调用其中的 MCP 工具。
5. 画布展示的是用户可见的 Agent 思维链/执行链和工具调用过程，而不是固定流程模板。

示例：
1. 用户问最新竞品促销，Agent 选择联网搜索 Skill，调用 `search-tavily` MCP。
2. 用户问指标口径，Agent 选择知识库 Skill，调用 Dify 知识库 MCP。
3. 用户问销量或库存，Agent 选择数据分析 Skill，后续调用 MySQL 查询 MCP。

### 6.3 意图识别

系统需识别以下意图：

| 意图 | 说明 | 右侧画布行为 |
| --- | --- | --- |
| 闲聊 | 问候、非数据对话 | 右侧保持空白或折叠 |
| 简单查询 | 查询单个指标数值 | 单 KPI 卡 + 简短结论 |
| 复杂分析 | 分析、诊断、查看原因 | KPI 组 + 图表 + 表格 + 结论 |
| 对比分析 | 区域、车型、时间对比 | 对比图表 + 排名表 |
| 风险预警 | 库存、异常、风险、超标 | 预警卡 + 风险原因 + 建议 |
| 指标解释 | “什么是”“定义”“口径” | 指标定义卡 |
| 联网搜索 | 需要外部市场/竞品信息 | 搜索结果卡 + 引用来源 |
| 追问 | 基于上下文继续提问 | Agent 自主补全上下文并选择后续 Skill |

### 6.4 动态 HTML 业务画布

用户看到的所有内容都应呈现在画布上。画布不应是固定模板，而应由 Agent 思维链/执行链的最终产物动态驱动渲染。画布优先流式展示大模型回答和 Agent 执行过程，随后根据最终结构化结果渲染 HTML 内容。

画布组件类型至少包括：
1. 单指标 KPI 卡。
2. 多指标 KPI 网格。
3. 折线图。
4. 柱状图。
5. 明细表格。
6. 结论与建议卡。
7. 风险预警卡。
8. 搜索结果卡。
9. 指标定义卡。

验收要求：
1. 闲聊问题不生成数据卡片。
2. “GS8 库存多少”类问题只展示必要卡片。
3. “分析/诊断”类问题展示完整组合。
4. 点击欢迎卡片触发回答后，画布必须正确渲染对应内容，不允许空白。
5. 画布组件不支持用户编辑或删除。
6. 明细表组件支持下载。
7. 画布模板需要后台可配置，后台配置位置和形态待设计。

### 6.5 Agent 思维链/执行链透明化

系统需向用户展示可理解的 Agent 思维链/执行链。该链路不是固定模板，而是每次由 ReAct Agent 根据问题、上下文、Skill 描述和工具返回动态生成。

可能出现的执行项包括：
1. 问题理解。
2. Skill 选择。
3. MCP 工具选择。
4. 知识库检索。
5. SQL 生成。
6. SQL 执行。
7. 联网搜索。
8. 数据解读。
9. 口径说明。
10. HTML 画布装配。

待确认：是否展示完整 SQL，还是默认折叠，仅允许有权限用户展开。

### 6.6 Mock API 优先策略

第一阶段后端接口全部先做 mock，目标是让原型中所有流程真实可跑：

1. mock 会话接口：新建会话、历史列表、历史详情、删除历史。
2. mock 流式问答接口：返回动态 Agent 思维链/执行链、Skill 选择、MCP 调用、SQL、回答文本、画布渲染数据。
3. mock 画布接口：返回 HTML 画布所需结构化 payload。
4. mock 下载接口：支持明细数据下载。
5. mock 联网搜索接口：先对接已跑通的 Tavily MCP 工具，也可临时返回固定搜索结果。
6. mock 数据资产接口：返回当前场景下可用表、字段、业务别名和指标口径。
7. mock Skill 注册接口：返回当前可用 Skill 及其 MCP 工具说明。

mock 阶段跑通后，再替换为真实接口：
1. MySQL 查询。
2. Dify 知识库检索。
3. Tavily MCP 联网搜索。
4. 内部大模型 SQL 生成与报告生成。
5. 权限系统。

### 6.7 Skill 与 MCP 工具体系

系统的核心扩展方式是 Skill + MCP：

1. Skill 是高阶业务能力说明，负责告诉 Agent：什么时候用、怎么用、需要哪些输入、会调用哪些 MCP、输出什么结构。
2. MCP 是实际工具层，负责执行具体能力，例如联网搜索、知识库检索、SQL 查询。
3. Agent 不直接写死调用链，而是根据问题和 Skill 描述自主选择。
4. 新增业务场景时，优先新增 Skill、MCP 工具和画布模板，而不是改固定流程。

已存在 MCP 示例：
1. `my-mcp-tool/search-tavily/`
2. MCP Server 名称：`tavily-search`
3. 工具函数：`search_web(query, search_depth="advanced", include_answer=True)`

说明：`search-tavily` 只是当前项目中已经跑通的 MCP 样例，用于提供后续 MCP 工具开发参考。后续所需 Skill 与 MCP 工具需要继续补充设计和实现，不能默认已经具备。

后续可能需要的 MCP：
1. Dify 知识库检索 MCP。
2. MySQL 查询 MCP。
3. 指标口径/元数据 MCP。
4. 文件下载或报表生成 MCP。

后续需要补充的 Skill：
1. 联网搜索 Skill：识别需要外部信息的问题，调用 Tavily MCP，并把搜索结果整理成画布可渲染结构。
2. 知识库检索 Skill：识别指标口径、车型别名、业务规则类问题，调用 Dify 知识库 MCP。
3. 数据查询 Skill：识别销量、库存、订单、排产等数据问题，调用 MySQL 查询 MCP。
4. 画布装配 Skill：根据回答、工具结果、表格数据和图表配置，生成业务画布 payload。
5. 下载 Skill：根据明细数据生成可下载文件或返回前端可下载结构。

Skill 编写要求：
1. 每个 Skill 必须写清楚适用场景、触发条件、输入参数、输出结构。
2. 每个 Skill 必须声明可调用的 MCP 工具及工具用途。
3. 每个 Skill 必须说明失败降级策略，例如 MCP 不可用时如何返回可理解提示。
4. 每个 Skill 的输出必须能被画布渲染层消费。
5. Skill 描述应足够清晰，让 ReAct Agent 能基于描述自主选择，而不是依赖硬编码路由。

### 6.8 SQL 与数据安全（MVP 后增强）

1. 仅允许生成和执行 SELECT 类查询。
2. 禁止 DROP、DELETE、UPDATE、INSERT、ALTER、TRUNCATE、CREATE 等危险操作。
3. Schema 组装阶段按用户 `allowed_tables` 过滤。
4. SQL 执行前二次校验引用表是否在用户权限内。
5. 数据分析节点必须使用广汽云自部署模型。
6. 无 Schema 或无权限时必须友好拒绝，不允许编造 SQL。
7. MVP 阶段暂不考虑数据权限问题。

### 6.9 知识库检索

1. Dify 仅作为知识库检索服务。
2. 默认使用混合检索 + Rerank。
3. 需支持车型、大区、国家、问答对、同环比等知识库。
4. 检索结果用于 SQL 生成、指标解释和报告口径补充。
5. 后续可将 Dify 知识库检索封装为 MCP 工具，由 Agent 通过 Skill 自主选择调用。

### 6.10 联网搜索

联网搜索已经有本地 MCP 工具 `my-mcp-tool/search-tavily/`：

1. MCP Server 名称：`tavily-search`。
2. 工具函数：`search_web(query, search_depth="advanced", include_answer=True)`。
3. 返回：Tavily JSON 搜索结果，保留中文。
4. 运行方式：stdio MCP。

产品行为：
1. 用户点亮“联网搜索”或问题明确包含最新资讯、网页、新闻、舆情、竞品动态时触发。
2. Agent 自主选择联网搜索 Skill，并调用 Tavily MCP。
3. 画布展示搜索结果卡和引用来源。
4. 如果搜索失败，应降级为仅基于内部数据回答，并提示外部搜索不可用。

### 6.11 会话与记忆

1. 支持历史会话列表。
2. 支持历史搜索。
3. 支持删除历史，并提供二次确认。
4. 删除全部历史后系统自动恢复空白新会话。
5. 支持用户偏好记忆，例如常查区域、车型、关注指标。

## 7. 当前业务场景与数据资产

产销存只是首个业务场景，不是产品唯一边界。后续可以扩展到财务、竞品、市场、售后等更多场景。

当前首个业务场景的数据资产来自 `DIFY/国际ChatBI-深度思考V3.0.yml` 中的 `table_promt_new` 与 `create_table_prompt_new` 变量，包含以下 8 张 MySQL 表：

| 表名 | 中文说明 | 主要用途 |
| --- | --- | --- |
| `v_dm_sal_wolesale_terminal_dly` | 批发终端日表 | 批发量、终端量、区域/国家/车型分析 |
| `v_dm_sal_sc_order_dly` | SC 订单日表 | 新增订单、区域/国家/车型订单分析 |
| `v_dm_sal_stock_dly` | 库存日表 | 总库存、经销商在途/在店、子公司库存、厂端库存、库龄 |
| `v_dm_sal_remain_order_dly` | 剩余订单日表 | 剩余订单、配车、未配车、超时预警 |
| `v_dm_sal_scheduling_dly` | 排产日表 | 交付量、下线量、排产执行 |
| `v_dm_sal_wolesale_terminal_dly_v2` | 批发终端预实表 | 批发/终端实际与计划 |
| `v_dm_sal_sc_order_dly_v2` | SC 订单预实表 | 订单实际与计划 |
| `v_dm_sal_scheduling_dly_v2` | 排产预实表 | 下线实际与计划 |

后续接入真实 MySQL 时，以上 DDL 和字段说明作为首批 Schema 语料。新业务场景应通过新增表说明、指标口径、别名词典和画布模板配置扩展，而不是改前端代码。

## 8. 技术架构约束

1. 前端：React 18 + Vite + ECharts。
2. 后端：FastAPI + WebSocket + REST。
3. 编排：LangGraph。
4. LLM：外部 DeepSeek 用于非敏感节点，内部自部署模型用于 SQL 和数据分析。
5. 知识库：Dify Retrieve API。
6. 业务数据库：MySQL，后续预留 StarRocks MCP。
7. 应用数据库：开发 SQLite，生产 PostgreSQL。
8. 联网搜索：Tavily MCP 工具。
9. 第一阶段接口：全部 mock。

## 9. 后台配置需求

后台配置暂未定型，但 PRD 先明确必须具备的配置对象：

1. 场景配置：场景名称、描述、可用表、指标口径、别名词典。
2. 表与字段配置：表名、中文名、字段名、字段中文含义、示例值、是否可查询。
3. Skill 配置：Skill 名称、适用场景、执行说明、可调用 MCP、输入输出结构。
4. MCP 工具配置：工具名称、连接方式、参数说明、返回结构、失败降级策略。
5. 画布模板配置：不同意图、不同场景下可使用的 HTML 组件组合。
6. 下载配置：哪些结果可下载、下载字段、文件命名规则。
7. 搜索配置：是否允许联网搜索、搜索深度、结果数量、是否展示引用。
8. 权限配置：用户角色、可访问场景、可访问表、可下载字段，MVP 暂不实现。

待设计：后台配置是独立管理页面、配置文件，还是数据库表。

## 10. 关键指标

待脑暴确认：
1. 首屏/悬浮窗打开速度。
2. 简单查询响应时间。
3. 深度分析响应时间。
4. Skill 选择准确率。
5. MCP 调用成功率。
6. 用户采纳率或复制/导出次数。
7. ETL 工单减少比例。

## 11. 待脑暴确认的问题
