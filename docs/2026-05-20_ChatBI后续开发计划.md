# ChatBI 后续开发计划

> 日期：2026-05-20  
> 基准文档：`docs/ChatBI_PRD_草案.md`  
> UI 基准：`前端原型/index_v3_dynamic.html`  
> 资产基准：`前端原型/assets/`  
> 当前结论：项目处于 MVP mock 骨架阶段，尚未达到原型级还原，也尚未完成 PRD 中定义的全部 mock 接口。

## 1. 当前阶段判断

当前 React + FastAPI 版本已经完成了第一版工程骨架：

1. 前端已建立 React 18 + Vite + ECharts 应用。
2. 后端已建立 FastAPI mock 服务。
3. 已有 SSE 流式问答接口。
4. 已有基础会话历史持久化。
5. 已有基础 Skill 注册表和数据资产接口。
6. 已有结构化画布组件渲染。
7. 已有明细 CSV 下载。
8. 已补入 SQL 校验异常与自愈演示节点。

但从 PRD 与原型验收角度看，当前只能算“MVP 技术骨架跑通”，不能算“MVP 原型复刻完成”。下一阶段必须先停止自创 UI，回到原型逐项复刻。

## 2. 必须纠偏的开发原则

1. `前端原型/index_v3_dynamic.html` 是唯一视觉与交互基准。
2. React 版不得重新设计工作台 UI，不得新增原型外的按钮文案、布局结构和视觉语言。
3. `前端原型/assets/` 中的头像与图标必须作为第一优先级资产来源。
4. 卡片历史与可访问数据必须按原型实现为跳出工作区边框的全局 fixed 浮层，而不是被工作区裁切或嵌在右侧画布内部。
5. 用户最终业务答案统一在右侧画布输出；左侧仅保留用户问题、Agent 执行链、SQL 折叠和 Skill/审计信息。
6. SQL 默认折叠，节点详情只展示用户可理解的执行摘要和可复核参数。
7. 后续截图统一放在 `qa/`。

## 4. Mock 接口完成度

PRD 中定义的 mock 接口包括：

| PRD mock 能力 | 当前接口 | 完成度 | 说明 |
| --- | --- | --- | --- |
| 新建会话 | `POST /api/conversations` | 70% | 可创建，但前端流程还需按原型新会话行为复刻 |
| 历史列表 | `GET /api/conversations` | 70% | 有列表，缺历史搜索与原型样式 |
| 历史详情 | `GET /api/conversations/{id}` | 60% | 可恢复基础内容，右侧画布恢复仍需加强验证 |
| 删除历史 | `DELETE /api/conversations/{id}` | 70% | 有删除和二次确认，需复刻原型弹窗 |
| 流式问答 | `POST /api/chat/stream` | 60% | 已有 SSE，但事件类型和 payload 还需标准化 |
| 非流式问答 | `POST /api/chat` | 40% | 可返回结果，主要用于调试 |
| 画布 payload | 目前内嵌在 chat result | 40% | 尚未独立画布接口，组件协议未文档化 |
| 明细下载 | `GET /api/downloads/mock-detail.csv` | 40% | 目前是 CSV，不是原型口径中的 Excel/xlsx 体验 |
| 联网搜索 | 目前在 chat mock 中返回固定结果 | 30% | 尚未形成独立 mock 搜索接口，也未真实调用 Tavily MCP |
| 数据资产 | `GET /api/config/data-assets` | 30% | 只有表级简表，缺字段、别名、指标口径与搜索字段 |
| Skill 注册表 | `GET /api/config/skills` | 40% | 有基础列表，缺输入输出结构、失败降级策略和 MCP 说明 |

结论：mock 接口没有全部写完。当前优先级不是继续堆真实 API，而是先把“原型交互 + mock 契约 + 画布 payload”定稳。

## 5. 下一阶段开发路线

### 阶段 A：原型级 UI 复刻收敛

目标：React 页面肉眼效果与 `index_v3_dynamic.html` 高度一致。

任务：

1. 将原型 CSS 变量、工作区尺寸、边距、圆角、阴影、滚动条、断点迁移到 React。
2. 用原型 DOM 结构重写主要布局：`chat-launcher`、`workspace-container`、`chat-pane`、`bi-canvas`。
3. 统一使用 `client/public/assets/` 中从原型复制来的头像和图标。
4. 顶部按钮、画布按钮、历史按钮全部改为 FontAwesome/原型图标风格。
5. 重做卡片历史和可访问数据为 workspace 外部 fixed drawer。
6. 可访问数据补齐原型中的搜索框、场景分组、字段行、标签。
7. 每次 UI 修复后用浏览器截图放到 `qa/`，与原型截图对比。

验收：

1. 精简窗口下卡片历史与可访问数据从工作区右侧跳出，不被边框裁切。
2. 全屏状态下两个浮层位置仍与原型一致。
3. 首页欢迎区、执行链、画布卡片、输入区、按钮图标与原型保持一致。

### 阶段 B：Mock API 补全与契约稳定

目标：后端 mock 覆盖 PRD 中第一阶段所有流程。

任务：

1. 定义 SSE 事件协议：`conversation_id`、`step`、`sql`、`answer`、`canvas`、`done`、`error`。
2. 增强 `step` payload：节点标题、工具、耗时、状态、输入、输出、错误、自愈记录。
3. 增强数据资产接口，返回表、字段、中文名、示例值、业务别名、指标口径。
4. 增强 Skill 注册接口，返回适用场景、触发条件、输入输出结构、MCP 工具、失败降级策略。
5. 拆出或文档化画布 payload 协议，至少覆盖 answer、kpi、chart、table、risk、definition、search_results、insight。
6. 下载接口改为 xlsx 或明确第一版 CSV，同时前端提示文案与原型保持一致。
7. 联网搜索 mock 拆出固定结果结构，为后续 Tavily MCP 替换留接口。

验收：

1. PRD 6.6 七类 mock 能力均有接口或明确 payload。
2. 欢迎卡三条黄金问题均能稳定生成执行链、SQL、最终回答和右侧画布。
3. 刷新/恢复历史后右侧画布不丢失。

### 阶段 C：黄金演示问题打磨

目标：先把 Demo 体验做稳，而不是泛化所有问题。

建议先固定 4 条演示问题：

1. `诊断中东公司 GS8 车型本月的库存异常并给出建议`
2. `分析本月中东公司车型排产与实际到货差额`
3. `GS8 当前库存是多少，是否超过安全线`
4. `什么是库存周转天数，计算公式是什么`

每条问题都要有：

1. 明确意图识别。
2. 不同执行链节点。
3. 不同画布组件组合。
4. 合理 mock 数据。
5. 可恢复历史。
6. 可下载明细。

### 阶段 D：真实接口替换准备

替换顺序建议：

1. 会话接口：SQLite mock 存储升级为可迁移存储结构。
2. 数据资产/字段口径：从 mock 配置升级为 YAML/JSON 或数据库配置。
3. Dify 知识库检索：先封装 Retrieve API，再封装为 MCP。
4. MySQL 查询：先只读 SELECT，增加 SQL 安全校验。
5. Tavily MCP 搜索：替换固定 mock 搜索结果。
6. LangGraph Agent：替换当前硬编码 intent/mock engine。

## 6. 近期执行顺序

1. 先做原型复刻修正，尤其是卡片历史、可访问数据、图标和工作区布局。
2. 再补数据资产与 Skill mock 结构，不急着接真实 API。
3. 固化画布 payload 协议，避免前后端互相猜字段。
4. 跑通并截图验证四条黄金问题。
5. 最后整理接口替换清单，进入真实服务接入阶段。

## 7. 风险与注意事项

1. 现在最大的风险是 UI 继续偏离原型，因此所有前端改动必须先对照 `index_v3_dynamic.html`。
2. 现在 mock engine 仍偏硬编码，后续要逐步配置化，但不要在 UI 复刻完成前大改架构。
3. 卡片历史和可访问数据不能放进 canvas 内部，否则精简窗口一定会出现裁切或定位偏差。
4. 可访问数据需要从“表级列表”升级到“字段级资产目录”，否则不满足 PRD 的数据资产透明化。
5. 下载第一版若继续用 CSV，需要在计划中明确，避免用户以为已经是 Excel xlsx。

## 8. 下一次开发检查清单

1. `client/public/assets/` 是否与 `前端原型/assets/` 同步。
2. React 中是否还存在原型外的可见文案。
3. 卡片历史和可访问数据是否是 fixed 全局浮层。
4. 是否所有截图都放在 `qa/`。
5. `npm run build` 是否通过。
6. 后端 `python -m compileall app` 是否通过。
7. 四条黄金问题是否全部有截图验证。
