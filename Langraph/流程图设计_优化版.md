# ChatBI LangGraph 流程图设计 (优化版)

这份文档基于对原流程图的 10 次逻辑推演，修复了原图中的安全漏洞、上下文绕过问题、缺失的短路机制，以及大模型解读数据的容错降级设计。

## 流程图 (Mermaid)

您可以安装 Markdown Preview Mermaid 插件（VS Code）或者将下方的代码复制到 [Mermaid Live Editor](https://mermaid.live/) 查看高清图表。

```mermaid
graph TD
    %% ================= 节点定义与样式 =================
    classDef init fill:#f9f9f9,stroke:#d3d3d3,stroke-width:2px,border-radius:5px
    classDef process fill:#e1f5fe,stroke:#81d4fa,stroke-width:2px,border-radius:5px
    classDef check fill:#fff9c4,stroke:#fff59d,stroke-width:2px,border-radius:5px
    classDef know fill:#e8f5e9,stroke:#a5d6a7,stroke-width:2px,border-radius:5px
    classDef danger fill:#ffebee,stroke:#ef9a9a,stroke-width:2px,border-radius:5px
    classDef terminal fill:#ffccbc,stroke:#ffab91,stroke-width:2px,border-radius:5px

    START((START)):::init
    END_NODE(((END))):::terminal
    
    init_node[初始化状态<br/>initialize_node]:::init
    intent_node[意图识别<br/>intent_node]:::process
    chat_node[闲聊回答<br/>chat_node]:::process
    context_node[上下文补全/指代消解<br/>context_answer_node]:::process
    task_node[构建分析任务<br/>build_bi_task_node]:::process
    support_check[意图与数据校验<br/>supportability_check_node]:::check
    support_resp[反问/拒绝支持<br/>supportability_response_node]:::danger
    
    knowledge_node[检索业务知识<br/>retrieve_knowledge_node]:::know
    schema_node[组装数据库Schema<br/>schema_context_node]:::know
    
    sql_gen[生成 SQL<br/>generate_sql_node]:::process
    sql_guard[SQL安全门禁<br/>guard_sql_node]:::check
    sql_repair[自我修正 SQL<br/>repair_sql_node]:::process
    sql_exec[执行 SQL<br/>execute_sql_node]:::process
    
    data_analyze[数据解读与图表配置<br/>analyze_data_node]:::process
    metric_exp[附加口径说明<br/>metric_explanation_node]:::process
    fail_node[全局失败兜底<br/>fail_node]:::danger

    %% ================= 核心链路流转 =================
    START --> init_node
    init_node --> intent_node
    
    %% 意图路由
    intent_node -. is_chat .-> chat_node
    intent_node -. is_followup .-> context_node
    intent_node -. is_data .-> task_node
    
    chat_node --> END_NODE
    
    %% 修复 1：上下文补全后，必须流向构建任务，避免绕过支持度校验！
    context_node --> task_node
    
    %% 校验与拒绝
    task_node --> support_check
    support_check -. unsupported .-> support_resp
    support_check -. supported .-> knowledge_node
    support_resp --> END_NODE
    
    %% 知识与结构检索
    knowledge_node --> schema_node
    %% 修复 4：如果找不到 schema，直接短路拒绝，别去让大模型瞎编 SQL
    schema_node -. schema_not_found .-> support_resp
    schema_node -. schema_found .-> sql_gen
    
    %% SQL 生成与安全拦截
    sql_gen -. success .-> sql_guard
    sql_gen -. error .-> fail_node
    
    sql_guard -. safe .-> sql_exec
    %% 修复 2：如果判定为恶意 SQL（DROP等），绝不去修，直接熔断！
    sql_guard -. malicious (恶意/越权) .-> fail_node
    sql_guard -. repairable .-> sql_repair
    
    %% 修复 3：修复闭环，明确必须流回安全门禁，且有次数限制 (比如 10 次)
    sql_repair -. retry_allowed (次数<10) .-> sql_guard
    sql_repair -. retry_limit_exceeded (次数超限) .-> fail_node
    
    %% 执行与解读
    sql_exec -. success .-> data_analyze
    sql_exec -. sql_error .-> sql_repair
    sql_exec -. fatal_error .-> fail_node
    
    %% 修复 5：解读降级容错，防大量数据撑爆模型
    data_analyze -. success .-> metric_exp
    data_analyze -. generation_failed (Token超限降级) .-> metric_exp
    
    metric_exp --> END_NODE
    fail_node --> END_NODE
```

## 主要架构改进亮点

1. **上下文补全前置化（防止绕过卫士）**：
   原图中，如果用户触发追问（followup），将补全上下文然后直接跳到查库并生成 SQL。新版设计中，`context_answer_node` 提取到了最前面（由意图节点路由）。一旦它把问题补全（例如“那华东区呢” -> “华东区的销售额是多少”），数据流会重新汇入 `build_bi_task_node` 并经受合规性校验。
2. **恶意 SQL 熔断机制**：
   在 `guard_sql_node` 加入了一条专门通往 `fail_node` 的红线。如果用户使用 Prompt 注入，诱导模型生成 `DELETE` 或者尝试查阅敏感表，安全门禁绝不尝试“修复（repair）”它，而是果断拦截并报错。
3. **闭环修复上限控制**：
   明确了 `repair_sql_node` 的输出必定流回 `guard_sql_node` 以确保安全。同时，为了防止反复陷入死循环，在边逻辑里增加 `retry_limit_exceeded` 判断（您的系统中可将其变量设为 10）。
4. **无 Schema 时的短路设计 (Fail Fast)**：
   如果在 `schema_context_node` 阶段发现提取不到任何有效的表，不要再傻傻地交给 `generate_sql_node` 让模型凭空捏造 SQL，而是直接跳转到 `supportability_response_node` 告诉用户“我不懂这块业务”。
5. **大规模数据容错降级 (Graceful Degradation)**：
   如果 `execute_sql_node` 查出了十万行数据，传给 `analyze_data_node` 必然会因为 Context 过大而崩溃。此时，如果分析失败，系统会触发 `generation_failed` 分支跨越错误，直接流向 `metric_explanation_node`。用户至少能看到一张包含原始数据的表格和指标定义，而不是一整页的 500 系统报错。
